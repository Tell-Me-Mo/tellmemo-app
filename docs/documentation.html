Deployment Options

TellMeMo supports multiple deployment strategies based on your needs:

Option 1: Docker Compose (Recommended)

Best for: Development, small teams, on-premise deployments

âœ… Advantages

  • Simple one-command setup
  • All services containerized
  • Easy to update and maintain
  • Built-in service health checks
  • Persistent data volumes
# Clone repository
git clone https://github.com/the-harpia-io/pm_master_v2.git
cd pm_master_v2

# Configure environment
cp .env.example .env
nano .env  # Add your API keys

# Start all services
docker-compose up -d

# Check status
docker-compose ps

Resource Requirements:

  • RAM: 8GB minimum (16GB recommended)
  • Disk: 20GB minimum
  • CPU: 4+ cores recommended

Option 2: Hybrid (Docker + Local Apps)

Best for: Development, debugging, custom modifications

ğŸ’¡ How It Works

  • Infrastructure services run in Docker (databases, etc.)
  • Backend runs locally with Python
  • Frontend runs locally with Flutter
  • Easier for development and debugging
# Start infrastructure only
docker-compose up -d postgres qdrant redis elasticsearch logstash kibana

# Terminal 1: Run backend
cd backend
python3 -m venv venv
source venv/bin/activate
pip install -r requirements.txt
uvicorn main:app --reload

# Terminal 2: Run frontend
flutter run -d chrome --web-port 8100

Option 3: Production Deployment

Best for: Production workloads, enterprise deployments

Recommended Architecture:

  • Database: Managed PostgreSQL (Supabase, AWS RDS, Google Cloud SQL)
  • Vector DB: Qdrant Cloud or self-hosted with backups
  • Backend: Docker container on Hetzner VPS, AWS EC2, or Google Compute Engine
  • Frontend: Static hosting (Firebase Hosting, Vercel, Netlify, Cloudflare Pages)
  • Monitoring: Sentry for errors, Langfuse for LLM observability
# Production Docker Compose
docker-compose -f docker-compose.prod.yml up -d --build

# With managed database
docker-compose -f docker-compose.prod.yml \
  -e POSTGRES_HOST=your-supabase-db.com \
  -e POSTGRES_PORT=5432 \
  up -d

Option 4: Kubernetes (Not Available)

âš ï¸ Kubernetes Not Currently Supported

Kubernetes deployment manifests are not included in the current release. For enterprise Kubernetes deployments, you'll need to create your own manifests based on the Docker Compose configuration.

Planned for future release: Helm charts and k8s manifests will be added in v2.0

# Copy environment file

# Edit .env and add your Anthropic API key

Installation Steps

Step 1: Clone Repository

git clone https://github.com/the-harpia-io/pm_master_v2.git
cd pm_master_v2

Step 2: Configure Environment

# Copy example configuration
cp .env.example .env

# Edit .env file
nano .env

Add your Anthropic API key and other required variables:

ANTHROPIC_API_KEY=sk-ant-api03-your-api-key-here
POSTGRES_PASSWORD=your-secure-password
REDIS_PASSWORD=your-redis-password
LANGFUSE_SECRET=$(openssl rand -hex 32)
LANGFUSE_SALT=$(openssl rand -hex 32)

Step 3: Start Docker Infrastructure

# Start all services
docker-compose up -d

# Check status
docker-compose ps

Step 4: Set Up Backend

cd backend

# Create virtual environment
python3 -m venv venv
source venv/bin/activate

# Install dependencies
pip install -r requirements.txt

# Start backend
uvicorn main:app --reload --host 0.0.0.0 --port 8000

Step 5: Set Up Frontend

# Get dependencies
flutter pub get

# Run application
flutter run -d chrome --web-port 8100

âœ… Installation Complete!

Your TellMeMo instance is now running at:

  • Frontend: http://localhost:8100
  • Backend API: http://localhost:8000/docs
  • Langfuse: http://localhost:3000

Common Use Cases

Real-world examples of how teams use TellMeMo based on actual API capabilities:

1. Portfolio-Level Reporting

Scenario

Executive needs a weekly summary of all projects across the portfolio to present to the board.

Solution:

  • Create Portfolio hierarchy with multiple Programs and Projects
  • Upload weekly meeting transcripts, status reports, and emails to each Project
  • Generate Portfolio-level Executive Summary (aggregates all child content)
  • Export as PDF for board presentation

API Endpoint: POST /api/v1/summaries/generate

{
  "summary_type": "executive",
  "hierarchy_id": "portfolio-abc-123",
  "hierarchy_type": "portfolio",
  "include_child_content": true
}

2. Risk Detection & Tracking

Scenario

PM needs to identify and track all risks mentioned across project meetings.

Solution:

  • Upload all meeting transcripts to the Project
  • AI automatically extracts risks during content processing
  • Query: "What are all the risks mentioned in this quarter?"
  • Generate Technical Summary focusing on risk mitigation

API Endpoint: POST /api/v1/queries

{
  "query": "What are all the risks mentioned in Q4 meetings?",
  "project_id": "project-xyz-456",
  "use_rag": true,
  "conversation_id": null
}

3. Multi-Turn Context-Aware Q&A

Scenario

Team member wants to have a conversation about decisions made in past meetings.

Solution:

  • Start conversation: "What was decided about the API redesign?"
  • Follow-up: "Who raised concerns about backward compatibility?"
  • Follow-up: "When is the migration deadline?"
  • System maintains conversation context across all queries

First Query:

POST /api/v1/queries
{
  "query": "What was decided about the API redesign?",
  "project_id": "project-xyz-456",
  "conversation_id": null  // Creates new conversation
}

Follow-up Query:

POST /api/v1/queries
{
  "query": "Who raised concerns about it?",
  "project_id": "project-xyz-456",
  "conversation_id": "conv-789"  // Maintains context
}

4. Real-Time Meeting Transcription

Scenario

Team wants live transcription during remote meetings.

Solution:

  • Connect to WebSocket: ws://localhost:8000/ws/audio
  • Stream audio chunks in real-time
  • Receive transcription updates as they're processed
  • Automatically save transcript to Project when meeting ends

WebSocket Channels:

  • /ws/audio - Real-time audio streaming & transcription
  • /ws/jobs - Job status updates (processing, embedding, summary generation)
  • /ws/notifications - User notifications
  • /ws/support-tickets - Support ticket updates

5. Cross-Project Insights

Scenario

CTO wants to understand how authentication is being discussed across all engineering projects.

Solution:

  • Query at Program or Portfolio level
  • Ask: "How are different teams approaching authentication?"
  • RAG searches across all child Projects
  • Provides unified answer with citations from each Project

API Endpoint: POST /api/v1/queries

{
  "query": "How are teams approaching authentication?",
  "program_id": "program-engineering-001",
  "search_child_projects": true,
  "use_rag": true
}

6. Automated Action Item Tracking

Scenario

Scrum master needs to track all action items and task assignments from sprint meetings.

Solution:

  • Upload sprint meeting transcripts
  • AI automatically extracts tasks with assignees and due dates
  • View all tasks via API: GET /api/v1/tasks?project_id={id}
  • Update task status as work progresses

Task Extraction (Automatic):

// When content is uploaded, tasks are auto-extracted
POST /api/v1/content
{
  "project_id": "project-xyz-456",
  "content_type": "transcript",
  "content_text": "John will update the API docs by Friday..."
}

// System automatically creates:
{
  "task": "Update API docs",
  "assigned_to": "John",
  "due_date": "2025-01-10",
  "status": "pending"
}

âš ï¸ Error Message

# Check status
